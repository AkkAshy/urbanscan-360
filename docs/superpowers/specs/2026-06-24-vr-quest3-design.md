# VR-режим для Meta Quest 3 — дизайн

**Проект:** urbanscan-360 · **Дата:** 2026-06-24 · **Автор:** Аня (mac-claude) · **Статус:** на ревью

---

## 1. Цель

Управляющий надевает Meta Quest 3, открывает отсканированный объект в браузере шлема и в иммерсивном VR:
- ходит по 360°-туру через хотспоты (стрелки-переходы),
- крутит головой, осматривает помещение,
- создаёт и удаляет хотспоты прямо в шлеме контроллерами.

## 2. Требования (зафиксированы в брейншторме)

- **Доставка:** браузерный WebXR в составе urbanscan-360 (A-Frame), НЕ нативка.
- **Объём:** просмотр + навигация по хотспотам + редактирование (создание/удаление связей) в VR.
- **Железо:** Meta Quest 3, ввод — контроллеры (лучи).
- **Без регрессий:** десктопный режим (мышь + HTML-UI) продолжает работать как сейчас.

## 3. Не-цели (YAGNI)

- Smooth-locomotion / WASD внутри сцены — бессмысленно: точка съёмки фиксирована, перемещение = телепорт между фото.
- Загрузка фото, создание/управление папками в VR — остаётся на десктопе.
- Hand-tracking — только контроллеры (руки потом, если понадобится).
- Мультиплеер, аннотации, timelapse, сравнение по датам — вне скоупа.
- Нативное Quest-приложение — это **fallback-подход B**, если браузерный WebXR (A) не даст приемлемого UX/перформанса.

## 4. Текущее состояние и затык

Вьювер живёт в оверлее `UploadPage.tsx` (`viewerOpen`). Сцена — `AFrameScene` (`<a-sky>` + `<a-camera>`), raycaster на `rayOrigin: mouse`. Управление — **HTML-оверлеи** (Tailwind `absolute`): кнопки закрыть/связи, нижняя панель выбора фото в `LinkEditor`, `ThumbnailStrip`, счётчик.

**Затык:** в иммерсивном WebXR HTML поверх canvas **не виден** — пользователь видит только содержимое A-Frame сцены. Внутри сцены интерактивны только стрелки `LinkArrows`, и ввод завязан на мышь.

Ключевые файлы: `components/viewer/AFrameScene.tsx`, `LinkArrows.tsx`, `LinkEditor.tsx`, `pages/UploadPage.tsx` (оверлей), `store/viewerStore.ts`.

## 5. Архитектура (подход A)

Минимальный VR-слой поверх текущего вьювера:

```
            ┌─ desktop: cursor rayOrigin:mouse ──► .clickable
raycaster ──┤
            └─ VR: laser-controls (2 руки) ──────► .clickable
                                                      │
                         ┌────────────────────────────┼─────────────────┐
                         ▼                            ▼                  ▼
                   LinkArrows (есть)            VRMenu (new)      VRPhotoPicker (new)
                   навигация/удаление        переключение/режимы   выбор цели связи
```

- **Ввод:** контроллеры Quest через A-Frame `laser-controls` + raycaster по `.clickable`. Триггер контроллера A-Frame транслирует в событие `click` — та же модель, что у мыши.
- **Навигация:** переиспользуем `LinkArrows` (стрелки уже `.clickable`).
- **Управление:** HTML невидим в VR → новые **in-scene** компоненты `VRMenu` и `VRPhotoPicker`.
- **Десктоп:** без изменений. HTML-UI рендерится при `!vrActive`, VR-UI — при `vrActive`.
- **Чистота:** новые VR-сущности оформляются как A-Frame-компоненты (`AFRAME.registerComponent`), а не россыпь `createElement` в `useEffect`.

## 6. Компоненты

### 6.1 `AFrameScene` (правки)
- Смонтировать `VRControllers` (две руки) в сцену.
- Слушать `enter-vr` / `exit-vr` → выставлять `vrActive` в store.
- Сохранить `cursor: rayOrigin: mouse` для десктопа; raycaster контроллеров добавляется отдельными узлами рук.
- Зависит от: store (`setVrActive`).

### 6.2 `VRControllers` (новый)
- **Назначение:** две руки Quest с видимым лучом и raycaster по `.clickable`.
- **Реализация:** `<a-entity laser-controls="hand: left/right" raycaster="objects: .clickable; far: 100">` + линия луча. Опц. `<a-entity blink-controls>`/визуал.
- **Зависит от:** наличие классов `.clickable` на интерактивных объектах.

### 6.3 `VRMenu` (новый, in-scene)
- **Назначение:** управление в VR (HTML-аналог невидим).
- **Состав:** `‹ ›` переключение фото, счётчик `N / M`, toggle **Связи** (editMode), кнопка **Создать**, **Выход из VR**.
- **Размещение:** HUD — закреплён перед камерой снизу, на комфортном расстоянии (~1.5–2 м), крупный текст. (Решено: HUD, не запястье.)
- **Каждый пункт:** `.clickable` плоскость + `a-text`, обработчик `click`.
- **Зависит от:** store (`currentIndex`, count, `linkEditMode`, действия `next/prev`, `setLinkEditMode`, `setVrPlacing`, выход).

### 6.4 `VRPhotoPicker` (новый, in-scene)
- **Назначение:** выбор целевого фото при создании связи — VR-замена нижней HTML-панели `LinkEditor`.
- **Состав:** дуга превью-плоскостей (`a-image` с thumbnail) доступных фото (кроме текущего и уже связанных), `.clickable`.
- **Появляется:** когда `vrPlacing != null`.
- **Действие:** клик по превью → `createPhotoLink(currentId, targetId, yaw, pitch)` → `fetchLinks` → сброс `vrPlacing`.
- **Зависит от:** `photos`, `links`, `vrPlacing`, `createPhotoLink`.

### 6.5 `skyPointToYawPitch` (новый util, вынести из `LinkEditor`)
- **Назначение:** `intersection.point {x,y,z}` → `{ yaw: 0..360, pitch }`. Формула уже зашита в `LinkEditor.tsx:56-62` — выносим в `utils/`, переиспользуют десктоп-`LinkEditor` и VR-создание (DRY).

### 6.6 `LinkArrows` (переиспользуем)
- Уже `.clickable` → луч ловит без изменений. Возможные правки: увеличить хитбокс и добавить hover-подсветку для комфортного прицеливания лучом.

## 7. Состояние (`viewerStore`)

Добавить:
- `vrActive: boolean` — в иммерсивном VR ли (из `enter-vr`/`exit-vr`).
- `vrPlacing: { yaw: number; pitch: number } | null` — ждём выбор цели связи.
- экшены `setVrActive`, `setVrPlacing`.

`UploadPage`: HTML-UI рендерим при `!vrActive`; `VRMenu`/`VRPhotoPicker` — при `vrActive`.

## 8. Потоки

- **Вход:** кнопка `vr-mode-ui` (A-Frame, уже `enabled`) на Quest → `enter-vr` → `vrActive=true` → VR-UI активен, HTML скрыт.
- **Навигация:** луч на стрелку → триггер → `goToId` → `fetchLinks` → новая сфера + стрелки. Фейд ~150 мс на переходе.
- **Переключение без хотспотов:** `VRMenu ‹ ›` → `store.prev/next`.
- **Создать связь:** `VRMenu` «Создать» → `vrPlacing` ждёт → луч на проход в `<a-sky>` → триггер → `skyPointToYawPitch` → `vrPlacing={yaw,pitch}` → `VRPhotoPicker` → клик по превью → `createPhotoLink` → `fetchLinks` → стрелка появилась.
- **Удалить связь:** `VRMenu` toggle «Связи» (editMode) → луч на стрелку → триггер → `deletePhotoLink` → `fetchLinks`.
- **Выход:** `VRMenu` «Выход» или системный жест → `exit-vr` → `vrActive=false` → HTML вернулся.

## 9. Края и ошибки

- Нет доступных фото для связи → `VRPhotoPicker` показывает текст «все привязаны».
- Снятие шлема / потеря трекинга → A-Frame `exit-vr` обрабатывает, возвращаемся на десктоп-UI.
- Большое фото грузится → лёгкий лоадер/фейд, не блокируем сцену.
- Десктоп без WebXR → кнопка VR не показывается (A-Frame сам), поведение как сейчас.
- **Комфорт:** без smooth-движения; мгновенный телепорт + фейд; меню на комфортной дистанции, крупный текст — против укачивания и промахов лучом.

## 10. Тестирование

⚠️ **WebXR требует HTTPS** (secure context) — заход со шлема по `http://<IP>` VR не активирует. Поэтому тестируем через задеплоенный фронт:

- **На реальном Quest 3:** правка → деплой `vercel --prod` из `frontend/` → открыть **https://urbanscan360.vercel.app** в браузере Quest 3 → «Enter VR». Прод-фронт ходит на прод-бэк `https://urban.saribek.uz`. Проверено вживую: фронт `200`, бэк `/api/auth/me/` → `401`, CORS preflight → `200` — HTTPS + живой бэк + CORS уже на месте.
- **Быстрые итерации без шлема:** Immersive Web Emulator (Chrome-расширение) на `localhost` (secure context) — эмулирует контроллеры Quest. Основная отладка тут.
- **Чек-лист:** вход/выход VR · навигация лучом · создание связи · удаление связи · десктоп-регрессия (мышь/HTML как раньше).
- Канат верифицирует вехи на реальном Quest 3 после деплоя фазы.

> Цикл «деплой ради теста на шлеме» медленноват (Vercel auto-deploy пока не настроен — деплой ручной). Поэтому основная отладка — в эмуляторе, на Quest — проверка вех. Опционально позже: https-туннель (cloudflared/ngrok) на локальный dev для VR-теста без деплоя.

## 11. Грубые фазы (вход для плана)

1. **Контроллеры + навигация** — войти в VR, ходить лучом по существующим стрелкам. Минимальный играбельный срез.
2. **VRMenu** — переключение фото, счётчик, выход.
3. **VR-редактирование** — `vrPlacing` + `VRPhotoPicker`, создание и удаление связей.
4. **Полировка** — фейды переходов, hover-подсветка, комфорт, прогон на Quest.

## 12. Решения и заметки

1. ✅ **Бэк при VR-тесте:** тестируем через задеплоенный Vercel-фронт (HTTPS) → прод-бэк `urban.saribek.uz`. Локальный бэк для VR-теста не нужен.
2. ✅ **VR-меню:** HUD снизу перед глазами (не запястье).
3. 📌 **Fallback:** если UX/FPS браузерного WebXR (A) не устроит — переключаемся на нативный подход B (Unity, возможно папка `UrbanScanVR`).
